# 原型页面功能需求说明（详细功能点版）

> 版本：V3（细化增强）  
> 基线：当前工作区原型页面  
> 目标：每个页面、每个功能点均可直接指导开发与测试执行。

---

## 1. 文档说明

### 1.1 适用范围
- 根目录原型页面（以“认定工作流程”主壳为准）。
- 包含：数据面板、项目管理、认定流程全链路、资料档案、订单入口。

### 1.2 功能点描述结构（统一模板）
每个功能点必须包含以下内容：
1. **功能目标**：该功能解决什么问题。  
2. **输入**：用户输入、页面参数、状态数据、外部依赖。  
3. **处理逻辑**：主流程、分支逻辑、校验逻辑、状态写入规则。  
4. **输出**：页面变化、状态变化、反馈信息、跳转结果。  
5. **用户交互流程**：用户从进入到完成的步骤。  
6. **异常与边界**：失败条件、拦截规则、提示文案口径。  
7. **验收标准**：可验证的通过条件。

## 2. 全局能力（所有页面适用）

### G-01 侧栏导航与路径一致性
- **功能目标**：确保用户在任意页面可快速切换模块，且定位不混乱。
- **输入**：点击一级/二级/三级菜单；当前页路由。
- **处理逻辑**：
  1. 一级菜单支持展开/收起。
  2. 子菜单点击后跳转对应页面。
  3. 当前页面对应菜单高亮（active）并保持上级展开。
- **输出**：页面跳转成功，侧栏状态与当前页面一致。
- **用户交互流程**：点击菜单 -> 页面加载 -> 继续操作。
- **异常与边界**：目标页面不存在时应提示而非静默失败（当前原型部分为占位）。
- **验收标准**：每个页面加载后，侧栏高亮项必须与页面标题一致。

### G-02 统一提示反馈（Toast）
- **功能目标**：对所有关键操作提供即时反馈，避免用户不确定。
- **输入**：保存、提交、上传、校验失败等事件。
- **处理逻辑**：生成提示文案 -> 顶部弹出 -> 自动关闭（约1.5~2s）。
- **输出**：用户可见成功/失败/警告信息。
- **用户交互流程**：触发动作 -> 看到结果 -> 决策下一步。
- **异常与边界**：连续触发时允许后发覆盖或队列展示（当前原型为单条临时提示）。
- **验收标准**：每个阻断场景必须有明确中文提示文案。

## 3. 页面：`数据面板.html`

### DB-01 站点分布地图
- **功能目标**：展示广西区域认定站点分布与容量概览。
- **输入**：
  - 地图边界JSON（广西）。
  - 站点数据（城市、经纬度、站点数量）。
- **处理逻辑**：
  1. 页面加载后初始化地图实例。
  2. 拉取地图数据并注册。
  3. 渲染站点散点，按数量设置视觉大小。
  4. 悬浮显示站点名称与数量。
- **输出**：可缩放可浏览的站点地图。
- **用户交互流程**：打开页面 -> 地图加载 -> 鼠标悬浮查看。
- **异常与边界**：网络失败时显示“地图加载失败，请检查网络连接”。
- **验收标准**：至少14个城市点可见；悬浮tooltip信息正确。

### DB-02 快捷入口
- **功能目标**：减少跨模块跳转成本。
- **输入**：点击卡片（项目准备、认定流程、系统设置）。
- **处理逻辑**：
  - 项目准备 -> 跳转 `项目管理首页.html`
  - 认定流程 -> 跳转 `认定流程首页.html`
  - 系统设置 -> 占位提示
- **输出**：成功跳转或占位提示。
- **交互流程**：点击卡片 -> 执行动作 -> 页面变化。
- **异常与边界**：占位功能必须给可识别提示，防止误判系统卡死。
- **验收标准**：前两项100%跳转可达；系统设置固定提示。

### DB-03 项目完成情况图
- **功能目标**：展示项目状态结构占比。
- **输入**：总项目数、已完成数、进行中数。
- **处理逻辑**：饼图渲染 + 图例 + tooltip占比。
- **输出**：图表与图例一致的状态分布。
- **交互流程**：进入页面自动渲染 -> 悬浮查看。
- **异常与边界**：数据为空时至少展示空态（当前原型为演示常量）。
- **验收标准**：各扇区比例与指标值匹配。

### DB-04 流程健康度（新增）
- **功能目标**：快速识别哪个阶段拖慢整体推进。
- **输入**：五阶段完成率（0~100）。
- **处理逻辑**：柱状图展示，低值按预警色显示。
- **输出**：阶段完成率排名与差异。
- **交互流程**：查看图表 -> 定位低分阶段 -> 进入对应页面处理。
- **异常与边界**：异常值（<0或>100）应在实现阶段做归一化校验。
- **验收标准**：五阶段均显示，标签与百分比准确。

### DB-05 归档闭环监控（新增）
- **功能目标**：判断归档工作是否闭环。
- **输入**：已完成/待复核/待备份数量。
- **处理逻辑**：环图渲染 + 图例数量展示。
- **输出**：归档状态分布。
- **交互流程**：查看分布 -> 对待处理项追踪到归档页。
- **异常与边界**：总量为0时图表需可显示空态。
- **验收标准**：图例数量与扇区一致，颜色语义清晰。

### DB-06 清单执行质量（新增）
- **功能目标**：汇总理论与实操执行质量，突出风险项。
- **输入**：理论完成率、实操完成率、附件完整率、风险列表。
- **处理逻辑**：渲染三指标卡 + 风险标签（完整/缺失/待上传）。
- **输出**：质量总览与风险清单。
- **交互流程**：查看指标 -> 查看风险项 -> 定位整改页面。
- **异常与边界**：风险项为空时显示“当前无风险项”。
- **验收标准**：指标卡与风险标签在同屏可见、状态语义正确。

## 4. 页面：`项目管理首页.html`

### PM-01 项目列表与检索
- **功能目标**：让用户快速定位并管理目标项目。
- **输入**：搜索词、状态筛选值、项目列表数据。
- **处理逻辑**：条件过滤后刷新展示。
- **输出**：筛选后的项目卡片列表。
- **交互流程**：输入关键词/选状态 -> 结果更新。
- **异常与边界**：无结果时展示空状态与引导文案。
- **验收标准**：任意组合筛选结果可复现。

### PM-02 新增项目
- **功能目标**：创建新项目并进入后续准备流程。
- **输入**：项目名称、编号、经理等字段。
- **处理逻辑**：必填校验 -> 创建记录 -> 刷新列表。
- **输出**：新项目可见。
- **交互流程**：点击新增 -> 填写 -> 保存 -> 返回列表。
- **异常与边界**：缺失必填项阻断提交并提示。
- **验收标准**：成功新增后刷新仍可见（本地存储）。

### PM-03 订单转项目
- **功能目标**：将待接收订单快速转为执行项目。
- **输入**：订单项点击动作。
- **处理逻辑**：订单状态变更 + 生成项目记录。
- **输出**：订单区状态变化，项目区新增卡片。
- **交互流程**：选择订单 -> 点击接收 -> 查看项目。
- **异常与边界**：重复接收应避免重复创建。
- **验收标准**：一次点击只生成一个项目。

- **功能目标**：汇总材料/设备/考务三清单完成情况。

- **处理逻辑**：计算总完成项与完成率并显示。
- **输出**：认定前准备总览。
- **交互流程**：查看总览 -> 点击缺失项进入子页。
- **异常与边界**：子页状态缺失时按未完成处理。
- **验收标准**：总览数值与子页状态一致。

- **输入**：8项材料状态切换。

- **输出**：材料清单完成状态。
- **交互流程**：逐项确认 -> 返回总览。
- **异常与边界**：重复操作不应造成计数异常。
- **验收标准**：8项全完成后总览同步更新。

- **功能目标**：确保设备与现场条件达标。

- **输出**：设备清单完成状态。
- **交互流程**：上传/确认 -> 返回总览。
- **异常与边界**：未上传时不应标记为完成。
- **验收标准**：设备项状态可持久化并回显。

### PRE-STAFF-01 考务人员清单
- **功能目标**：完整记录四类考务人员信息。

- **处理逻辑**：增删改表格数据，类别状态与统计联动。
- **输出**：人员清单与完成类别数。
- **交互流程**：录入/导入 -> 勾选类别完成 -> 返回总览。
- **异常与边界**：关键字段为空时需阻断保存（实现阶段要求）。
- **验收标准**：类别完成状态与总览一致。

- **交互流程**：点击保存 -> 校验 -> 结果提示。
- **异常与边界**：部分上传时必须阻断。
- **验收标准**：附件不齐绝不允许保存成功。

### PRAC-CHECK-04 提交规则
- **功能目标**：确保提交数据完整与可审计。
- **输入**：点击提交按钮。
- **处理逻辑**：
  1. 校验20条是否全部勾选。
  2. 校验1-6附件齐全。
  3. 通过后提交并返回实操页。
- **输出**：提交成功并跳转，或失败提示。
- **交互流程**：点击提交 -> 双重校验 -> 成功返回。
- **异常与边界**：任一条件不满足必须阻断。
- **验收标准**：未满足条件无法提交。

- **异常与边界**：上传中按钮置禁用，避免重复提交。
- **验收标准**：6项均有上传与重传能力。

### ARCH-02 进度联动
- **功能目标**：实时反映归档完成程度。
- **输入**：6项上传状态布尔值。
- **处理逻辑**：计算完成数量，更新进度条和文案（0/6~6/6）。
- **输出**：归档进度可视化更新。
- **交互流程**：上传任意一项 -> 进度刷新。
- **异常与边界**：状态回退时进度同步回退。
- **验收标准**：进度数据与上传状态严格一致。

### ARCH-03 归档事项状态联动
- **功能目标**：让用户明确当前归档子任务状态。
- **输入**：已上传数量阈值。
- **处理逻辑**：按阈值更新a1~a4状态标签。
- **输出**：归档事项从“待归档/待生成”到“已归档/已生成”。
- **交互流程**：上传 -> 观察状态表变化。
- **异常与边界**：阈值变化必须单调正确，不能提前完成。
- **验收标准**：状态变化与阈值规则一致。

### ARCH-04 执行归档
- **功能目标**：触发归档闭环动作。
- **输入**：点击“执行归档”。
- **处理逻辑**：
  1. 先校验6项是否上传齐全。
  2. 不齐阻断；齐全则执行归档流程。
  3. 写入归档完成状态（本地存储）。
- **输出**：按钮显示“归档完成”，状态更新。
- **交互流程**：点击执行 -> 校验 -> 完成提示。
- **异常与边界**：重复点击需防抖或幂等处理。
- **验收标准**：未齐全时不能执行归档。

### ARCH-05 提交归档
- **功能目标**：结束本项目认定流程。
- **输入**：点击“提交归档”。
- **处理逻辑**：
  1. 校验6项已上传。
  2. 校验已执行归档。
  3. 通过后提交并跳流程首页。
- **输出**：流程闭环完成反馈。
- **交互流程**：点击提交 -> 校验 -> 成功跳转。
- **异常与边界**：缺任何前置条件都阻断并提示具体原因。
- **验收标准**：通过前置条件后必定跳转成功。
