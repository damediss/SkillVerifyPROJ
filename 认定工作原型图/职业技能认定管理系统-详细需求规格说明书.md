## 2. 产品背景与目标

### 2.1 业务背景

职业技能认定工作当前主要依赖线下纸质文档和分散的Excel表格管理，存在信息分散、流程割裂、材料分散、监控缺失、校验薄弱等痛点。需构建一套全流程线上化管理系统实现数字化转型。

### 2.2 产品目标

1. **项目全生命周期管理**：从订单到归档闭环管理
2. **认定流程标准化**：每个阶段操作规范、可追溯
3. **数据集中可视化**：通过数据面板实时掌握运营状况
4. **过程管控强约束**：关键节点设置强制校验，防止遗漏

### 2.3 核心业务链路

```
服务企业下单 → 创建项目 → 项目准备阶段（章程/成员/WBS/进度/风险 → 审核）
  → 认定筹备期（学员导入 → 10项资料收集 → 考评员配置 + 考场配置）
    → 认定前（材料清单 + 设备清单 + 考务人员清单，可选主管复核锁定）
      → 理论笔试（现场检查 + 成绩备份 + 合格反馈）
        → 实操考试（20项检查清单 + 评分记录 + 结果反馈）
          → 认定归档（6项文件上传 → 执行归档 → 提交归档）
            → 申领补贴（通过学员材料核对与导出，辅助流程，不阻断主线闭环）
```

## 4. 系统架构概览

### 4.1 技术栈建议

| 层级 | 技术选型建议 | 说明 |
|------|-------------|------|
| 前端框架 | **Vue 3**（落地首选） / React | 组件化开发；Vue 路线配套 **Ant Design Vue**，与设计规范（Ant Design）一致 |
| 状态管理 | **Pinia** / React Context | 管理全局状态与项目上下文；认定流程进度以后端为准，前端逐步替代 localStorage |
| 图表库 | ECharts 5.x | 数据面板图表、地图等 |
| 后端框架 | **Spring Boot 3**（落地首选） / Node.js | RESTful API，统一前缀 `/api/v1` |
| 运行时 | **JDK 17 或 21（LTS）** | 与 Spring Boot 3 对齐 |
| 数据库 | MySQL 8+ / PostgreSQL | 关系型存储；**个人实施阶段二选一即可**，不必双库并行 |
| 缓存 | Redis（可选） | 会话扩展、热点缓存；首期可暂缓，遇性能或会话需求再引入 |
| 文件存储 | 对象存储（OSS/**MinIO**） | 开发期推荐 **MinIO（S3 兼容）**，生产可换云 OSS |
| 前端工程化 | **Vite** / Webpack | 推荐 Vite + Vue 3 |

### 4.4 后端技术选型（落地建议）

以下为与个人实施场景匹配的**默认后端栈**，便于简历与面试表述一致；可在 README 中摘录。

| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | Spring Boot 3.x | Web MVC；单体优先，无需首期微服务化 |
| 安全 | Spring Security + **JWT** | 对齐第 16.4 条「JWT + 角色权限」迁移项；Refresh Token 可二期 |
| 持久化 | **MyBatis-Plus** | 清单、文档内容等可用 JSON 字段建模（MySQL JSON / PostgreSQL jsonb） |
| 迁移 | **Flyway**（或 Liquibase） | 表结构版本化，便于多设备拉代码后本地一键演进 |
| 校验 | Jakarta Bean Validation | 与统一错误响应（第 13 章）配合 |
| API 文档 | **springdoc-openapi**（Swagger UI） | 便于 Vue 联调与作品集展示 |
| DTO | 手写或 MapStruct | 规模扩大后再引入 MapStruct |
| 测试 | JUnit 5 + Mockito；可选 **Testcontainers** | 优先覆盖实操清单、归档闸门等关键规则 |
| 构建 / 容器 | Maven 或 Gradle；**Docker Compose** | 建议 Compose 编排数据库与 MinIO，降低换机成本 |

**首期可不引入：** 微服务拆分、消息队列、Elasticsearch（除非后续明确检索需求）。

### 4.6 源码管理与多设备开发约定

独立开发需在**办公电脑与个人电脑**等多设备间切换时，**必须使用 Git 托管源码**，避免拷贝目录导致覆盖与冲突。

| 约定 | 说明 |
|------|------|
| 远程仓库 | 使用 **GitHub / GitLab / Gitee** 等远程托管；每次告一段落执行 `commit` 与 `push`，换设备先 `pull` |
| 分支 | 单人可默认 `main`；功能较大时使用 `feature/*` 便于回溯 |
| 忽略提交 | 必须配置 `.gitignore`：`target/`、`node_modules/`、IDE 目录、本地密钥与 **含密码的 `application-*.yml`** |
| 机密配置 | 仓库内仅保留 `application-example.yml` 等模板；真实密码、JWT 密钥使用环境变量或本机私有配置文件（且不提交） |
| 合规 | 若公司设备限制外网或 Git，须遵守用人单位信息安全制度，不得违规外传代码 |

### 5.4 状态一致性规则（强约束）

**规则一：实操检查清单提交规则**
- 20条勾选未完成 → 禁止提交
- 序号1-6附件未上传齐全 → 禁止保存草稿与提交

**规则二：认定后归档规则**
- 6项文件未上传齐 → 禁止执行归档、禁止提交归档
- 未执行归档 → 禁止提交归档

**规则三：全局表单校验**
- 必填项缺失 → 阻断提交并提示具体缺失字段名
- 校验时机：失焦校验 + 提交时全量校验

#### 9.2.5 PR-04 考评员配置（筹备期新增）

**对应原型：** `考评员配置.html`、`考评员工作台.html`、（库）`考评员信息管理.html`

**业务目标：** 在一个项目中，将多名考评员加入同一个考评组，指定 1 名组长；配置成功后推送状态信息；从“实操题库”调取题目发放实操考评任务；考评员端创建/填写任务并提交；组长复核任务。

**功能需求：**

- **考评组配置**
  - 支持设置考评组名称（如：电工实操A组）
  - 支持配置多名考评员为组成员
  - 支持在组成员中指定 1 名为组长（单组模式，后续可扩展多组）
- **考评员来源**
  - 支持“手动新建考评员”
  - 支持“从考评员库调用”（调用后为任务侧副本，可再编辑，不影响库）
- **推送配置状态**
  - 配置完成后可对组内成员推送站内消息：项目编号、组名、角色（组长/成员）、进入工作台入口
- **实操任务发放（从题库调取）**
  - 题库数据结构需包含：科目、题目、时长、步骤（步骤名/分值/要点）
  - 发放时支持：选择题目、选择考评员（多选）、可选截止日期
  - 发放结果：为每位考评员生成一条“实操考评任务”，并推送消息
- **考评员端任务创建/填写**
  - 考评员端仅能查看分配给自己的任务
  - 支持开始填写（进入“填写中”）、保存进度、提交给组长复核
  - 提交后任务只读（不可再修改）
  - 若组长打回整改，任务进入“需整改”，允许整改并再次提交
- **组长复核**
  - 组长端可查看本组内“已提交”任务列表
  - 支持复核通过/打回整改，需填写复核意见（可选）
  - 复核结果通知对应考评员

**状态流转：**

```
assigned(已发放) → in_progress(填写中) → submitted(已提交) → reviewed(已复核通过)
                                        └→ rework(需整改) → in_progress → submitted …
```

**接口建议（V2正式后端实现）：**

```
POST /api/v1/projects/:id/evaluator-groups
Body: { groupName, leaderId, memberIds[] }

POST /api/v1/projects/:id/notifications/push
Body: { targetUserIds[], type: 'config'|'task'|'review', message }

GET  /api/v1/question-bank/practical?subject=&page=1&pageSize=20

POST /api/v1/projects/:id/practical-tasks/dispatch
Body: { questionId, dueDate?, evaluatorIds[] }

GET  /api/v1/practical-tasks?mine=1&status=&page=1
PUT  /api/v1/practical-tasks/:taskId/progress
Body: { stepScores: Record<string, number>, note? }

POST /api/v1/practical-tasks/:taskId/submit
POST /api/v1/practical-tasks/:taskId/review
Body: { action: 'approve'|'rework', note? }
```

**对应原型：** `考评员配置.html`、`考评员工作台.html`

**能力清单：**

1. 项目内组队：多名考评员进入同一个考评组，指定 1 名组长
2. 配置推送：配置成功后推送状态信息给组内考评员
3. 任务发放：从“实操题库”调取题目，发放考评任务给考评员
4. 任务执行：考评员创建/填写实操考评任务（按详细科目步骤/要点/分值）
5. 组长复核：组长复核实操考评任务，支持通过/打回整改
6. 权限访问：考评员端仅能查看“我的任务”；组长额外可见“复核面板”

**非功能约束：**

- 组长仅允许 1 人（单组模式）
- 任务只读策略：提交后任务进入只读；复核通过后只读；打回整改后恢复可编辑

**状态字段（V2.2原型调整）：**

- 物料库考评员状态与项目考评员配置状态统一为：`认定任务中` / `认定筹备中` / `空闲`
- 原型规则（建议）：
  - 发放实操任务后，将被分配的考评员状态置为 `认定任务中`
  - 任务全部复核通过后，可自动回流为 `空闲`（正式版本建议由后端根据任务状态实时计算）

**数据分桶（按项目隔离，V2.2原型实现）：**

- `prep_evaluator_config_by_project_v1`：按 `projectCode` 存储该项目的考评员配置列表
- `prep_evaluator_group_by_project_v1`：按 `projectCode` 存储该项目的考评组元信息

## 14. 非功能需求

### 14.1 性能要求

| 指标 | 要求 | 测量方式 |
|------|------|----------|
| 首屏加载时间 | ≤ 3秒 | Lighthouse / 实际测量 |
| 列表渲染 | 1000条数据内无明显卡顿 | Chrome DevTools Performance |
| 图表渲染 | ≤ 2秒 | 从数据加载到渲染完成 |
| API响应时间(P95) | ≤ 500ms | APM工具 |
| 文件上传 | ≤ 10MB的图片≤3秒 | 模拟上传 |
| 并发用户 | ≥ 50并发 | 压测工具 |

### 14.2 可用性要求

- 所有阻断场景必须有中文提示文案
- 关键操作（删除、提交、归档）需二次确认弹窗
- 表单校验在失焦和提交时均触发
- 全局导航不超过3级深度
- 所有可点击元素需有hover状态反馈
- 断网时给出友好提示

### 14.3 兼容性要求

| 浏览器 | 最低版本 |
|--------|:--------:|
| Google Chrome | ≥ 90 |
| Microsoft Edge | ≥ 90 |
| Mozilla Firefox | ≥ 90 |

### 14.4 安全要求

| 要求 | 说明 |
|------|------|
| XSS防护 | 用户输入做HTML转义 |
| 敏感信息脱敏 | 身份证号、手机号在列表展示时脱敏 |
| 文件上传限制 | 仅允许图片(jpg/png)、PDF、Word、Excel |
| 文件大小限制 | 图片≤10MB，文档≤50MB |
| 防重复提交 | 按钮点击后disabled |
| SQL注入防护 | 参数化查询 |
| CSRF防护 | Token机制 |

### 14.5 可维护性要求

| 要求 | 说明 |
|------|------|
| 代码组织 | 前端按功能模块分包，后端分层（Controller/Service/DAO） |
| 状态管理 | 前端全局状态集中管理，localStorage统一key前缀 |
| API版本 | URL前缀版本控制 `/api/v1/` |
| 日志 | 关键操作记录操作日志（谁、何时、做了什么） |
| 错误码 | 统一错误码体系，前端按code做差异化处理 |

### 14.6 数据库要求

- 所有表必须有 `created_at` 和 `updated_at` 字段
- 采用软删除（加 `deleted_at` 字段）替代物理删除
- 索引策略：外键字段、查询频繁字段建索引
- JSON字段：MySQL 5.7+ / PostgreSQL 原生JSON支持

## 15. 验收标准

### 15.1 核心链路验收

| 编号 | 验收场景 | 通过条件 |
|:----:|----------|----------|
| AT-01 | 完整认定流程 | 从创建项目→准备文档→审核→筹备期→认定前→理论笔试→实操考试→归档，全链路走通 |
| AT-02 | 实操检查清单强制校验 | 20项未全勾选/附件未齐时，保存草稿和提交均被阻断 |
| AT-03 | 归档前置校验 | 6项文件未齐时，执行归档和提交归档均被阻断 |
| AT-04 | 数据面板数据准确 | 图表数据与各模块实际数据一致 |
| AT-05 | 项目CRUD | 创建、编辑、删除项目正常 |
| AT-06 | 学员CRUD | 新增、编辑、删除学员，资料导入状态可跟踪 |
| AT-07 | 审核流 | 准备文档提交→企业审核→内审→通过/驳回全流程 |
| AT-08 | 阶段流转 | 前一阶段完成后才能进入下一阶段 |
| AT-09 | 文件上传 | 文件上传/重传/状态回显正常 |
| AT-10 | 申领补贴材料核对 | 通过认定学员列表、M001～M006 齐备度计算、导出/演示操作有明确反馈；正式环境禁用纯演示入口 |

### 15.2 测试用例命名规范

格式：`{模块代码}-{功能点代码}-{序号}`

示例：
- `PRAC-CHECK-04-01`：提交规则-未全勾选阻断
- `ARCH-EXEC-01-01`：执行归档-前置校验失败
- `PM-DELETE-01-01`：删除项目-二次确认后删除

每个功能点最少3条用例：正常路径、校验失败路径、边界路径（空值/重复操作/回退）。

### 15.3 关键校验点测试矩阵

| 功能点 | 正常路径 | 校验失败 | 边界 |
|--------|----------|----------|------|
| 实操清单提交 | 20项全勾选+附件全→提交成功 | 20项未全勾选→阻断 | 已提交后再次提交→提示 |
| 实操草稿保存 | 1-6附件全→保存成功 | 附件部分缺失→阻断 | 无任何勾选时保存→阻断 |
| 归档执行 | 6文件全上传→执行成功 | 文件未全上传→阻断 | 已执行后再次点击→已禁用 |
| 归档提交 | 已执行归档→提交成功 | 未执行归档→阻断 | 已提交后再次提交→已禁用 |
| 项目删除 | 确认删除→删除成功 | 取消→不删除 | 删除后列表不再显示 |
| 学员删除 | 确认删除→删除成功 | 取消→不删除 | 删除后关联资料一并清除 |
| 新增项目 | 必填项完整→创建成功 | 必填项缺失→阻断 | 项目编号重复→提示 |
| 新增订单 | 必填项完整→提交成功 | 必填项缺失→阻断 | 人数填0或负值→阻断 |

### 16.7 个人开发里程碑与排期参考

以下排期基于 **§1.5《实施背景说明（个人开发场景）》** 所述 **每周约 10 小时** 有效开发；全量功能接近说明书第 6～11 章时，总时长往往长于 MVP。**总工期量级：** MVP 约 **4.5～6 个月**，接近全量（含考评员任务闭环、数据面板完整能力等）约 **7～10 个月**，可按实际周工时线性缩放。

#### 16.7.1 每周时间切分（建议）

| 块 | 时长（约） | 内容 |
|----|------------|------|
| A | 3h | 后端：表结构/接口或领域规则（当周单一主题） |
| B | 4h | 前端：1～2 个页面或一条纵向链路对接 |
| C | 2h | 联调、边界与验收点对齐（第 15 章） |
| D | 1h | 提交记录与 README/CHANGELOG 简短更新（便于作品集叙述） |

原则：**每周只推进一个可演示的垂直切片**（接口 + 页面 + 联调），避免同时大改架构与赶进度。

#### 16.7.2 依赖顺序（与功能章节对应）

1. **地基**：数据库迁移、统一响应/校验、文件上传、认证与角色占位（第 5、13、14 章）。  
2. **主干**：订单 → 项目 → 工作台 → 五项准备文档与双审（第 7、8 章）。  
3. **流程**：阶段进度后端化 → 筹备期学员与 10 项资料 → 认定前 → 理论 → **实操强校验** → **归档闸门**（第 9 章）。  
4. **增强**：物料库（第 11 章）、考评员组与实操任务闭环（第 9.2.5、9.2.7、11.5 节）、数据面板聚合（第 6 章）、资料档案只读（第 10 章）、申领补贴（§9.7，可与归档结果联动）。

#### 16.7.3 按月里程碑（MVP 优先，约第 1～6 月）

| 月份 | 目标摘要 | 建议演示产出 |
|:----:|----------|----------------|
| 1 | 地基 + 首个列表链路 | 登录（或占位）、项目/列表分页真实接口 |
| 2 | 项目 / 订单 / 工作台 | 下单 → 转项目 → 进入工作台 |
| 3 | 五项准备文档 + 双审 | 任一文档从编辑到通过/驳回闭环 |
| 4 | 阶段进度 + 筹备期 | 学员与 10 项资料、阶段解锁条件可用 |
| 5 | 认定前 + 理论 | 三清单与主管复核锁定；理论 KPI 与清单 |
| 6 | 实操 + 归档 | **AT-01 / AT-02 / AT-03** 级场景可演示 |

第 7 个月起：物料库、考评员任务与题库、数据面板完整图表、广西地图精细实现等按优先级穿插。

#### 16.7.4 MVP 范围建议（可延后项）

为控制周期，下列可在 MVP 之后迭代：**广西地图 Leaflet/ECharts 精细实现**、**数据面板总经理/指挥台视图**、**批量导入学员与申报材料自动生成**、**站内消息实时推送**（可先记录通知列表）、考评员状态实时计算的复杂策略、**申领补贴正式导出与监管对接（§9.7）**等。**不建议削减：** 实操 20 项与附件强校验、归档 6 项与执行/提交前置校验、阶段解锁规则 —— 与第 15 章高风险验收直接相关。

#### 16.7.5 作品集与面试叙事要点

- **技术栈一句话：** Vue 3 + Vite + Pinia + Ant Design Vue + Spring Boot 3 + MyBatis-Plus + Flyway + MySQL 或 PostgreSQL + MinIO（S3 兼容）+ JWT；可选 Redis；Docker Compose 本地编排。  
- **可展开的设计决策：** 阶段状态以后端为准；实操与归档「前端体验校验 + 后端权威校验」双重保障；文件类型与大小限制统一在文件服务层落实（第 14 章）。
